### `public.guides` RLS ポリシー

```sql
ALTER TABLE public.guides ENABLE ROW LEVEL SECURITY;

-- 全員 read 可
CREATE POLICY "guides_public_read" ON public.guides
  FOR SELECT USING (true);

-- 認証済みユーザーが自分の行のみ INSERT
CREATE POLICY "guides_insert_self" ON public.guides
  FOR INSERT TO authenticated
  WITH CHECK (auth.uid() = user_id);

-- 自分の行のみ UPDATE / DELETE
CREATE POLICY "guides_update_self" ON public.guides
  FOR UPDATE TO authenticated
  USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id);

CREATE POLICY "guides_delete_self" ON public.guides
  FOR DELETE TO authenticated
  USING (auth.uid() = user_id);
```

### `public.travelers` テーブル

```sql
CREATE TABLE public.travelers (
  id          bigint PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  user_id     uuid NOT NULL UNIQUE REFERENCES auth.users(id) ON DELETE CASCADE,
  name        text NOT NULL,
  country     text NOT NULL,
  interests   text[] NOT NULL DEFAULT '{}',
  created_at  timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX travelers_user_id_idx ON public.travelers(user_id);
```

`user_id` に **UNIQUE 制約** あり = 1ユーザー1旅行者プロファイル。重複 INSERT は Postgres エラーコード `23505` で `app/actions/travelers.ts` 側でユーザーフレンドリーなメッセージに変換。

RLS は **SELECT も含めて全アクション「自分の行のみ」**。旅行者一覧の公開エンドポイントなし（プライバシー優先）。

```sql
ALTER TABLE public.travelers ENABLE ROW LEVEL SECURITY;

CREATE POLICY "travelers_select_self" ON public.travelers
  FOR SELECT TO authenticated USING (auth.uid() = user_id);
CREATE POLICY "travelers_insert_self" ON public.travelers
  FOR INSERT TO authenticated WITH CHECK (auth.uid() = user_id);
CREATE POLICY "travelers_update_self" ON public.travelers
  FOR UPDATE TO authenticated
  USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id);
CREATE POLICY "travelers_delete_self" ON public.travelers
  FOR DELETE TO authenticated USING (auth.uid() = user_id);
```

⚠️ **guides と travelers の整合性**: 両方とも user_id UNIQUE で 1ユーザー1プロファイル統一済み（commit 73d2453 で guides にも UNIQUE 追加）。

### `public.messages` テーブル

```sql
CREATE TABLE public.messages (
  id           bigint PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  sender_id    uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  recipient_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  body         text NOT NULL CHECK (length(body) > 0 AND length(body) <= 2000),
  created_at   timestamptz NOT NULL DEFAULT now(),
  CONSTRAINT messages_no_self_message CHECK (sender_id <> recipient_id)
);
CREATE INDEX messages_pair_idx ON public.messages (sender_id, recipient_id, created_at);
CREATE INDEX messages_recipient_idx ON public.messages (recipient_id, created_at DESC);
ALTER PUBLICATION supabase_realtime ADD TABLE public.messages;
```

RLS:
- SELECT: 参加者 (sender or recipient = auth.uid()) のみ
- INSERT: sender = auth.uid() のみ
- UPDATE/DELETE ポリシーなし → メッセージは編集不可・削除不可（証跡保持）

`supabase_realtime` publication に追加済み。クライアントは `supabase.channel("chat-A-B").on("postgres_changes", ...)` で INSERT を購読。

### Supabase Auth - **ユーザー手動設定が必要**

MCP からは Auth 設定変更不可。ダッシュボードでの手動操作が必要：

1. **Authentication → URL Configuration**
   - Site URL: `https://nomadomo.vercel.app`
   - Additional Redirect URLs: `http://localhost:3000/**`, `https://nomadomo.vercel.app/**`
2. **Authentication → Sign In / Up**
   - 開発中は **「Confirm email」を OFF** 推奨（オンだとメール到着＆クリックまでログイン不可）

`auth.users` テーブルは Supabase 管理。アプリ側からは `supabase.auth.getUser()` でアクセス。

### キー実装ノート

**`app/page.tsx`（ホーム）**
- `'use client'` の SPA。`screen` state で 4 画面（home/profile/chat/myprofile）切替。
- `useMemo(() => createClient(), [])` でブラウザクライアント生成。
- `supabase.auth.onAuthStateChange` でログイン状態を `userEmail` state に同期。
- TopBar に「+ ガイドになる」(`/guides/new`)、ログイン状態に応じて😊アバター or 「ログイン」リンク。
- myprofile 画面下部に `<form action={signout}>` でログアウトボタン（ログイン時のみ）。

**`app/actions/auth.ts` / `app/actions/guides.ts`**
- `'use server'`
- 戻り値: `{ error?: string }` または `{ errors?: Record<field, string> }`（useActionState 用）
- 成功時は `redirect('/')`（next/navigation）。`redirect()` は throw するので `useActionState` の戻り値型は型上ちょっとゆるい。
- `revalidatePath('/', 'layout')` でレイアウト再検証。

**`app/guides/new/page.tsx`（Server Component）**
- `await createClient()` → `getUser()` → 未ログインなら `redirect('/login?next=/guides/new')`
- 認証済みなら `<GuideForm userEmail={user.email}/>` を返す

**`app/guides/new/guide-form.tsx`**
- emoji 10択: 🧑 🍜 ⛩ 🎨 🌙 🚲 🏯 ☕ 📷 🍶
- タグ5択: Food / Temples / Nightlife / Hidden / Art（ホームのフィルターと一致）
- 言語7択: EN / JP / ZH / KR / FR / ES / DE
- タグ/言語は React state で管理、選択値を `<input type="hidden" name="tags" value={t}/>` で複数 hidden 入力にして FormData の `getAll("tags")` で受け取り

**`proxy.ts`**
- 全リクエストで `supabase.auth.getUser()` を呼んで cookie リフレッシュ
- matcher: 静的アセットを除外

## 7. 認証フロー（実装詳細）

```
[Browser] → /signup フォーム submit
       ↓
[Server Action signup(formData)]
   - email/password バリデーション
   - supabase.auth.signUp({email, password})
   - error なら { error } 返却（useActionState で表示）
   - 成功なら revalidatePath + redirect('/')
       ↓
[proxy.ts] 以降の全リクエストで supabase.auth.getUser() を呼んでセッション更新
       ↓
[Browser] ホーム再描画。onAuthStateChange で userEmail state 更新
```

ガイド登録時：

```
[Browser] → /guides/new
       ↓
[Server Component] await createClient() → getUser()
   - user なし → redirect('/login?next=/guides/new')
   - user あり → <GuideForm userEmail={user.email}/>
       ↓
[Browser] フォーム入力 → submit
       ↓
[Server Action createGuide(formData)]
   - getUser() 再チェック（防御的）
   - フィールドバリデーション → errors 返却 or
   - supabase.from('guides').insert({ user_id, ...fields })
     - RLS により auth.uid() = user_id でのみ通る
   - revalidatePath('/') → redirect('/')
```

## 8. 既知の問題・ハマりポイント

### 環境
1. **Cowork のサンドボックスは Windows のファイルロックと相性悪い**。`.git/index.lock` が掴まれて削除不可になることあり。`mcp__cowork__allow_cowork_file_delete` 経由で削除可能。
2. **クロスプラットフォーム npm install 問題**: Windows で入れた node_modules に Linux 用 SWC バイナリがないため、サンドボックスで `next build` 実行すると Bus error。サンドボックスでは `npm install` 後 `tsc --noEmit` まで。
3. **Edit/Write ツールが大きめのファイルで時々途中で切れる**現象を観測。`cat > file << 'EOF'` のヒアドキュメントで bash 経由書き込みが回避策。
4. **Supabase 無料プロジェクトは 1-2 週間で自動 INACTIVE**。`mcp__3c675608…__restore_project` で復活可。

### 設計・コード
5. **既存3件のテストデータの user_id が NULL**。RLS の UPDATE/DELETE ポリシーでは誰も触れない。
6. **Vercel env var の Sensitive を ON にしてしまっている**。`NEXT_PUBLIC_*` には本来意味なし。OFF にすべき。
7. **`useActionState` の戻り値型と `redirect()` の組み合わせがやや無理筋**。`redirect()` が throw する仕様なので、TypeScript 上はリターン型が `Promise<undefined>` でも実行時には到達しない。バグはないが型上は曖昧。
8. **email/password のみ実装**。Magic link / OAuth (Google, Apple) は未実装。
9. **ガイドのプロフィール編集・削除画面が未実装**。RLS は通る用意あるがフロントなし。
10. **「Edit」ボタンが myprofile 画面に飾りで存在**。クリックしても何も起きない。
11. **マイプロフィール画面が "Alex Chen" ハードコード**。実ユーザーと無関係。要置き換え。

### 11.1 残課題（正直に書く）

- **画像はパブリック URL。** 個人写真を載せる場合は signed URL ＋ private bucket に切り替えるべき。今は `getPublicUrl()` でアクセス制御なし。
- **ImageUploader の × は即時 Storage 削除。** 「うっかり」リスクあるが、現状は素直で実装シンプル。後で confirm() か undo を入れてもよい。
- **Inbox 一覧の lastBody は realtime 連動なし。** `unreadByPeer` は realtime で増えるが、最終メッセージ本文・時刻は inbox を開き直
